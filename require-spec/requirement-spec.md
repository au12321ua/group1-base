# 需求分析

## Introduction

### Infomation manager

#### 原始需求

主要负责管理整个教学服务系统的用户、权限、课程等基础信息，并能够提供一定程度的系统安全性保证。用户主要分学生、教师、教务管理人员和系统管理员等。具体功能如下：
- 用户基本信息管理：教务管理人员能够添加、编辑、删除学生和教师用户，搜索与查看学生和教师信息，设置用户类别，管理用户权限，修改用户信息。
- 不同类型用户登录系统后的功能：可管理个人信息，填写或修改部分内容，上传照片等。
- 课程基本信息管理：包括课程名称，课程学分，课程容量，课程考核方式等基本信息。  
- 用户权限管理：主要体现在后续的各个子系统，不同类型用户在各个子系统中拥有不同类型的权限，系统能够保证用户不越权访问，其中系统管理员具有最高的权限。 系统安全管理负责对整个系统的安全性进行管理，包括用户注册信息的安全、用户密码的安全管理，防入侵管理，系统日志的记录与管理等。
  
#### 需求细化

1. 利益相关者：普通用户、管理员用户、本组开发人员、其他业务服务开发人员、项目验收者
2. 各方需求（细化）：
    - 普通用户（学生、教师）：
        - 登录与会话：
            - 支持账号密码登录，登录成功后获取access/refresh token。
            - 在access token有效期内无需重复登录；过期后可用refresh token无感续期。
            - 异常登录（密码错误、token失效）应给出明确错误提示，不暴露敏感信息。
        - 个人信息管理：
            - 可查看本人完整档案。
            - 仅可修改可编辑字段：联系方式、邮箱、紧急联系人、头像/照片。
            - 不可修改基础身份字段：学号/工号、姓名、性别、出生日期、入学/入职信息、角色。
            - 每次修改应记录变更时间与操作者。
        - 隐私与访问控制：
            - 仅本人和授权管理员可查看个人详细信息。
            - 非本人请求应默认拒绝，返回无权限错误。

    - 管理员用户（教务管理员）：
        - 管理员工作台：
            - 提供面向管理员的图形化界面，用于学生/教师信息检索、查看、新增、编辑、删除。
            - 支持按学号/工号、姓名、角色等条件组合查询。
            - 支持批量导入与批量删除（可先支持CSV导入）。
        - 账号与权限管理：
            - 可分配和调整普通用户角色与权限范围。
            - 可重置用户密码、禁用/启用账户。
            - 所有高风险操作（删除、批量导入、权限变更）必须写入审计日志。
        - 更严格鉴权：
            - 管理员密码策略高于普通用户。
            - 管理员access token有效期短于普通用户。

    - 本组开发人员：
        - 迭代目标：
            - 原型阶段优先完成核心流程：登录鉴权、用户管理、课程信息管理、权限控制、日志记录。
            - 设计需预留扩展点，便于后续新增角色、权限粒度、认证方式和数据源。
        - 可维护性：
            - 认证中心与信息管理模块职责分离，API边界清晰。
            - 关键配置（token时长、密码策略、日志级别）可配置化。
        - 可观测性：
            - 日志按级别（DEBUG/INFO/WARN/ERROR）输出。
            - 提供统一的日志查询入口或检索方式，便于定位问题。

    - 其他业务服务开发人员（系统内部调用方）：
        - 统一鉴权能力：
            - 提供可验证token的能力（含公钥获取），并可提取可信uid与role。
            - 业务服务通过网关透传身份信息后可直接做业务鉴权。
        - 标准化API：
            - 提供RESTful风格基础信息API（用户、角色、课程等）。
            - API应区分用户态调用与服务态调用，并按权限进行访问限制。
            - 对外提供稳定字段命名与错误码约定，避免频繁破坏性变更。
        - 数据一致性：
            - 明确“基础信息主数据”与“业务扩展数据”的归属边界。
            - 对重复存储字段提供同步策略（定时校验或事件通知）。
        - 数据提供能力：
            - 向排课业务组提供教师名单、待选课名单、校历、选课学生名单查询能力。
            - 向选课业务组提供培养方案查询能力（含课程ID列表）。
            - 名单类接口支持分页查询与CSV导出。
            - 输出结果需包含数据版本或快照时间，保证跨系统口径可追溯。

    - 项目验收者：
        - 验收范围：
            - 原始需求中用户、权限、课程、安全管理相关功能可演示并可验证。
            - 任何需求变更（如课程字段调整）需在设计文档中留痕并说明原因。
        - 验收材料：
            - 提供可运行系统、接口说明、测试用例与测试结果。
            - 提供典型业务场景演示脚本（普通用户、管理员、越权访问拦截）。

3. 非功能需求（NFR）：
    - 安全性：
        - 密码不得明文存储，采用带盐哈希。
        - token采用签名机制并支持过期控制，服务端可识别失效token。
        - 所有受保护接口必须进行鉴权与授权校验，默认拒绝越权访问。
        - 关键管理操作写入不可抵赖的审计日志（含时间、操作者、对象、操作类型、结果）。
    - 性能：
        - 在原型环境下，90%常规查询接口响应时间不超过500ms。
        - 登录接口在常规并发下响应时间不超过800ms。
        - 支持至少100个并发会话的基础读写能力（原型目标）。
    - 可用性与可靠性：
        - 单个服务实例可用性目标不低于99.0%（原型阶段）。
        - 出现异常时返回统一错误码与可读提示，不因单点异常导致系统全面不可用。
        - 数据库写操作需具备事务性，避免脏写和部分成功。
    - 可维护性：
        - 模块分层明确：鉴权中心、信息管理模块、网关适配层。
        - 核心接口需有文档（参数、返回、错误码、权限说明）。
        - 关键业务逻辑应覆盖基础自动化测试（单元测试/接口测试）。
    - 可扩展性：
        - 支持后续替换SQLite为其他关系型数据库，业务接口尽量保持不变。
        - 权限模型支持新增角色与细粒度权限点。
        - 认证机制预留扩展接口（如后续接入多因素认证）。
    - 可审计性：
        - 日志分级、可检索、可按时间和用户维度追踪。
        - 审计日志保留周期不少于180天（原型可先本地存储，后续可迁移集中日志系统）。
    - 易用性：
        - 管理员常见操作（查询、编辑、导入）应在3次点击内到达。
        - 表单校验与错误提示清晰，减少误操作成本。
    - 兼容性与部署：
        - 后端接口遵循HTTP/JSON规范，便于不同技术栈服务调用。
        - 在Windows/Linux开发环境下可完成本地部署与联调。
    - 数据口径与可追溯性：
        - 对外名单与方案类数据返回中必须包含snapshotTime或version字段。
        - 同一查询条件在同一数据版本下应返回一致结果。
        - 名单查询与方案查询请求应记录调用方、查询条件与返回版本号，便于审计追踪。
